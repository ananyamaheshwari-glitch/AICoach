
AI-Powered Quiz Application Documentation
1. System Overview
This project is a full-stack web application designed to administer quizzes and provide users with personalized, AI-generated feedback on their performance. The system evaluates user answers, identifies strengths and weaknesses, and generates a detailed report with a customized learning roadmap.

The application is built with a modern, lightweight tech stack:

Frontend: React (with Vite), Tailwind CSS, and Axios for API communication.
Backend: Node.js with the Express framework.
Database: SQLite for simple, file-based data persistence.
Authentication: Secure, cookie-based session management.
AI Service: Hugging Face Inference API for Large Language Model (LLM) based evaluation.
2. AI-Driven Development Lifecycle (AIDLC)
The development of this project followed a lifecycle tailored for integrating AI capabilities:

Planning & Scoping: The primary objective was defined: create a quiz platform where feedback is not static but dynamically generated by an AI to be personal and insightful. The scope was limited to a single quiz topic ("Cloud Computing") to prove the concept.

Data & Schema Design: A simple yet effective database schema was designed to support the application's core entities: users, questions, and quiz results. The quiz_results table was specifically designed to store not just the score but also the complete AI-generated report for future retrieval.

Backend API Development: A standard RESTful API was built using Node.js and Express. Endpoints were created for user authentication (login, logout, status check) and the core quiz functionality (fetching questions, submitting answers, retrieving results).

AI Workflow Integration: This was the most critical phase. A multi-step, chained-prompting strategy was designed to interact with the Hugging Face LLM. This breaks down the complex task of evaluation into smaller, more reliable steps, ensuring a high-quality, structured output.

Frontend Development: A responsive user interface was built with React. The frontend was designed to be a "thin client," consuming the backend API and focusing on presenting the data and user flow clearly, from login to results.

Testing & Refinement: The application was tested manually through a comprehensive QA process covering the full user journey. This included testing login, quiz functionality, and verifying the structure and quality of the AI-generated reports.

4. API Documentation
The backend exposes the following endpoints under the /api prefix.

Authentication (/api/auth)
POST /login

Description: Authenticates a user and creates a session.
Body: { "username": "testuser", "password": "password123" }
Response (Success): 200 OK with user object and a connect.sid session cookie.
Response (Failure): 401 Unauthorized for invalid credentials.
POST /logout

Description: Destroys the current user session.
Response: 200 OK on success.
GET /status

Description: Checks if the user has an active session. Used by the frontend on initial load.
Response (Authenticated): 200 OK with user object { "user": { "id": 1, "username": "testuser" } }.
Response (Not Authenticated): 401 Unauthorized.
Quizzes (/api/quizzes)
GET /questions/:topic

Description: Fetches all multiple-choice questions for a given topic.
Example URL: /api/quizzes/questions/cloud
Response: 200 OK with an array of question objects.
POST /submit

Description: Submits user answers for evaluation, triggers the AI workflow, and stores the result.
Body: { "user_answers": { "1": "A", "2": "C", ... } }
Response: 201 Created with the resultId and the complete report object.
GET /results/:resultId

Description: Retrieves a previously completed quiz result for the logged-in user.
Example URL: /api/quizzes/results/123
Response (Success): 200 OK with the complete report object.
Response (Failure): 404 Not Found if the result doesn't exist or doesn't belong to the user.
5. Database Schema
The application uses an SQLite database with three main tables:

users: Stores user credentials.

id: INTEGER PRIMARY KEY - Unique identifier for the user.
username: TEXT NOT NULL UNIQUE - The user's login name.
password: TEXT NOT NULL - The user's hashed password.
created_at: TIMESTAMP - Timestamp of user creation.
questions: Stores the quiz questions and answers.

id: INTEGER PRIMARY KEY - Unique identifier for the question.
question_text: TEXT NOT NULL - The text of the question.
option_a, option_b, option_c, option_d: TEXT NOT NULL - The multiple-choice options.
correct_option: TEXT NOT NULL - The correct answer ('A', 'B', 'C', or 'D').
topic: TEXT NOT NULL - The topic of the question (e.g., "Cloud").
quiz_results: Stores the outcome of each quiz attempt.

id: INTEGER PRIMARY KEY - Unique identifier for the quiz result.
user_id: INTEGER NOT NULL - Foreign key referencing the users table.
quiz_timestamp: TIMESTAMP - Timestamp of when the quiz was submitted.
score: REAL NOT NULL - The calculated score (0-100).
raw_answers: TEXT NOT NULL - A JSON string of the user's submitted answers.
final_report: TEXT - A JSON string containing the complete, AI-generated report object.
6. AI Workflow Explanation (3-Call System)
To ensure a high-quality and structured evaluation, the backend uses a three-step, chained-prompting strategy to communicate with the LLM.

Call 1: Initial Evaluation (evaluateAnswers)

Input: The backend first calculates the objective facts: the user's score, a list of topics they answered correctly (strengths), and a list of topics they answered incorrectly (weak_areas). This data is passed to the AI.
AI Task: The LLM is instructed to act as an expert evaluator. Its only tasks are to analyze the pattern of errors and generate two qualitative fields: error_pattern and key_insight.
Output: A structured JSON object containing the pre-calculated score, strengths, weak areas, and the AI-generated insight.
Call 2: Detailed Feedback (getDetailedFeedback)

Input: Based on the user's score from Call 1, the system chooses one of two paths:
Low Score (<50): The AI is given the user's weak_areas and prompted to act as a friendly tutor, explaining the core concepts in a simple, encouraging way.
High Score (>=50): The AI is given the user's strengths and prompted to act as a senior architect, providing advanced insights and real-world context.
Output: A formatted Markdown string containing the detailed textual feedback.
Call 3: Final Report Synthesis (generateFinalReport)

Input: The AI is given the full analysis from Call 1 and the detailed feedback text from Call 2.
AI Task: The LLM is instructed to act as an AI career coach. Its task is to synthesize all the provided information into a final, structured JSON report containing an overall_summary and a personalized_roadmap (an array of actionable next steps).
Output: The final JSON object for the report.
This multi-call approach isolates responsibilities, making the AI's job easier at each step and dramatically increasing the reliability and quality of the final report.

7. Security Design
Security is a primary consideration in the application's design.

Authentication: The system uses a robust, cookie-based session management system (express-session). It does not rely on insecure methods like storing tokens in localStorage.
Password Hashing: User passwords are never stored in plaintext. They are securely hashed using bcryptjs before being saved to the database.
Session Security: Session cookies are configured with httpOnly: true, which prevents client-side JavaScript from accessing them, mitigating XSS attacks. In a production environment, they would also be marked as secure: true to ensure they are only sent over HTTPS.
Data Segregation: API endpoints that retrieve sensitive data (like quiz results) are protected by middleware that ensures a user can only access their own data. The getResultById controller verifies that the user_id on the result matches the id of the logged-in user.
Environment Variables: Sensitive information, such as the HUGGINGFACE_HUB_TOKEN and session secret, is managed through a .env file and is not hardcoded into the application. The application will fail to start if the API token is missing, preventing accidental insecure deployment.
8. Testing Approach (Manual QA)
The application can be fully tested manually by following the standard user journey.

Setup: Run both the frontend and backend servers as per the project's setup instructions.
Login:
Navigate to the login page.
Attempt to log in with incorrect credentials and verify that an "Invalid credentials" error is shown.
Log in with the correct default credentials (testuser / password123) and verify redirection to the dashboard.
Dashboard:
Verify that the welcome message displays the correct username.
Click the "Start Quiz" button.
Quiz Page:
Verify that all questions are displayed correctly with their multiple-choice options.
Attempt to submit without answering all questions and verify that an error message appears.
Select an answer for every question and click "Submit Quiz".
Results Page:
Verify that you are redirected to the results page and a report is displayed.
Check that all sections of the report are present: Score, Key Insight, Strengths, Weak Areas, and the full AI Evaluation Report.
Check for any formatting issues or strange characters in the AI-generated text.
Session Persistence & Logout:
Refresh the results page and verify that the report re-loads correctly (fetching from the API).
Click the "Logout" button and verify you are redirected to the login page.
Attempt to access the dashboard or results page URL directly and verify you are redirected back to the login page.
9. Limitations and Future Improvements
While the application is fully functional, there are several areas for future enhancement.

Limitations
Single Quiz Topic: The application is currently hardcoded to only support the "Cloud Computing" quiz.
No Admin Interface: There is no UI for adding, editing, or deleting quiz questions. This must be done directly in the database.js seed script.
Basic Frontend State Management: The frontend uses simple useState and useEffect hooks. For a larger application, a more robust state management solution (like React Context or a dedicated library) might be necessary.
No Quiz History: Users cannot view a list of their past quiz attempts.
Future Improvements
Dynamic Topic Selection: Add a backend endpoint to fetch all available quiz topics and update the dashboard to allow users to choose from a list.
Admin Panel: Create a separate, secure admin interface for managing users and quiz content (questions and topics).
Quiz History Page: Implement a new page where logged-in users can see a list of their past quizzes, scores, and click to view the full report for each.
Loading Skeletons: Improve the user experience by adding loading skeleton UI elements on the Quiz and Results pages while data is being fetched from the API.
Global Error Handling: Refactor the backend to use a global error handling middleware in Express to centralize error logic and reduce code duplication in controllers.
